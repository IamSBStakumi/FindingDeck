# FindingDeck MVP仕様書

## 1. プロダクト概要

FindingDeck は、Trivy や govulncheck などのOSSセキュリティスキャナの結果を集約し、開発チームが脆弱性対応を運用しやすくする軽量SaaSである。

スキャン自体はFindingDeck側では実行しない。
ユーザーのCI上でスキャンを実行し、その結果JSONをFindingDeckにアップロードする。

FindingDeckはアップロードされたレポートをもとに、重要度別の件数、前回との差分、履歴、通知を提供する。

## 2. MVPの目的

MVPの目的は、以下の仮説を検証することである。

> 小規模開発チームは、OSSスキャナの結果をCI成果物として放置するのではなく、履歴・差分・通知として管理できるなら、継続的に使う価値を感じる。

MVPでは「高機能な脆弱性管理」ではなく、**スキャン結果をチームで見られる状態にすること**を最優先する。

## 3. 初期ターゲット

初期ターゲットは、GitHub Actions または GitLab CI を使っている小規模Web開発チーム。

想定ユーザーは以下。

- バックエンドエンジニア
- テックリード
- 小規模開発チームのCTO
- セキュリティ専任者がいない開発チーム
- 受託開発会社の開発責任者

## 4. MVPで解決する課題

MVPでは以下の課題に絞る。

- CIで生成した脆弱性レポートが見返されない
- 前回スキャンから悪化したのか改善したのか分からない
- HIGH / CRITICAL の新規発生に気づきにくい
- Slack / Discord に読みやすい形で通知できない
- 過去のスキャン履歴が残らない
- セキュリティ専任者がいないため、脆弱性対応が属人的になる

## 5. MVPの対応スキャナ

初期対応スキャナは以下の2つに限定する。

### Trivy

主に以下の用途を想定する。

- ファイルシステムスキャン
- コンテナイメージスキャン
- 依存パッケージの脆弱性検出

### govulncheck

主にGoプロジェクトの脆弱性検出を想定する。

MVPでは、npm audit、Semgrep、osv-scanner、Dependabot Alert などには対応しない。

## 6. MVPの主要機能

### 6.1 アカウント作成

ユーザーがFindingDeckにアカウントを作成できる。

MVPでは、認証方式はシンプルでよい。

- メールアドレスログイン
- GitHubログイン

どちらか一方でよい。

### 6.2 プロジェクト作成

ユーザーは、スキャン結果を管理する単位としてプロジェクトを作成できる。

プロジェクトには、以下のような情報を設定できる。

- プロジェクト名
- リポジトリURL
- 利用するスキャナ
- 通知先設定

### 6.3 APIキー発行

CIからFindingDeckへレポートをアップロードするためのAPIキーを発行できる。

MVPでは、1プロジェクトにつき1つのAPIキーがあれば十分。

必要な操作は以下。

- APIキーの発行
- APIキーの再生成
- APIキーの無効化

### 6.4 レポートアップロード

CIからTrivyまたはgovulncheckのJSONレポートをアップロードできる。

ユーザーは、GitHub Actions や GitLab CI で以下の流れを組む。

1. CI上でスキャナを実行する
2. JSONレポートを生成する
3. FindingDeckへアップロードする
4. FindingDeck上で結果を確認する

アップロード時に最低限受け取る情報は以下。

- スキャナ種別
- レポートファイル
- ブランチ名
- コミットSHA
- 実行日時

### 6.5 スキャン結果一覧

プロジェクトごとに、過去のスキャン結果を一覧表示できる。

一覧では以下を確認できる。

- スキャン日時
- スキャナ種別
- ブランチ名
- コミットSHA
- CRITICAL件数
- HIGH件数
- MEDIUM件数
- LOW件数
- 前回比

### 6.6 最新スキャン結果のダッシュボード

プロジェクトの最新状態をダッシュボードで確認できる。

表示する内容は以下。

- 最新スキャン日時
- 総Finding数
- 重要度別件数
- 新規発生件数
- 解消済み件数
- 継続中件数
- 前回比

MVPでは、グラフは必須ではない。
まずは数字と一覧で十分。

### 6.7 Finding一覧

検出されたFindingを一覧表示できる。

一覧で確認できる項目は以下。

- 脆弱性ID
- パッケージ名
- 現在のバージョン
- 修正バージョン
- 重要度
- スキャナ種別
- 初回検出日
- 最終検出日
- 状態

状態は最低限、以下の3つでよい。

- Open
- Resolved
- Ignored

### 6.8 Finding詳細

各Findingの詳細を確認できる。

詳細画面では以下を表示する。

- 脆弱性ID
- タイトル
- 説明
- 重要度
- 対象パッケージ
- 現在のバージョン
- 修正バージョン
- 参照URL
- 検出されたスキャン履歴
- 状態

MVPでは、修正方法の自動提案は不要。

### 6.9 前回との差分表示

最新スキャン結果と前回スキャン結果を比較し、差分を表示する。

差分は以下に分類する。

- New：新規に検出されたFinding
- Existing：継続して検出されているFinding
- Resolved：今回検出されなくなったFinding
- Reopened：一度解消済みだったが再発したFinding

MVPでは、差分表示が最重要機能である。

### 6.10 Slack通知

スキャン完了後、条件に応じてSlackへ通知できる。

MVPで通知する内容は以下。

- プロジェクト名
- スキャン日時
- スキャナ種別
- CRITICAL件数
- HIGH件数
- 新規発生件数
- 解消済み件数
- レポート詳細URL

通知条件は最低限、以下でよい。

- CRITICALが1件以上ある場合
- HIGHが1件以上ある場合
- 新規Findingがある場合
- 毎回通知

### 6.11 Discord通知

Slackと同様に、Discord Webhookにも通知できる。

ただし、MVPではSlackを優先し、Discordは後回しでもよい。

### 6.12 Ignore設定

対応不要なFindingをIgnoredにできる。

MVPでは、以下の操作ができればよい。

- FindingをIgnoredにする
- Ignore理由を記録する
- Ignoredを解除する

期限付きIgnoreはMVPでは不要。

## 7. MVPでやらないこと

MVPでは以下は対象外とする。

- FindingDeck側でのスキャン実行
- GitHub App連携
- GitLab App連携
- 自動Pull Request作成
- AIによる修正提案
- SBOM管理
- コンプライアンス帳票出力
- SSO
- 組織・チームの複雑な権限管理
- 請求書払い
- 複数ロール管理
- Webhook以外の高度な通知連携
- npm audit対応
- Semgrep対応
- osv-scanner対応
- Dependabot Alert連携
- Jira / Linear / GitHub Issue連携

## 8. ユーザー利用フロー

### 初回セットアップ

1. ユーザーがFindingDeckに登録する
2. プロジェクトを作成する
3. APIキーを発行する
4. Slack通知先を設定する
5. CIにスキャン実行とアップロード処理を追加する
6. CIを実行する
7. FindingDeck上で結果を確認する

### 継続利用

1. CIで定期的にスキャンが実行される
2. レポートがFindingDeckにアップロードされる
3. FindingDeckが前回との差分を算出する
4. 必要に応じてSlackへ通知される
5. ユーザーが新規Findingを確認する
6. 対応不要なものはIgnoredにする
7. 修正後、次回スキャンでResolvedになる

## 9. MVPの成功条件

MVPの成功条件は以下。

- 自分のGo/Next.jsプロジェクトで継続利用できる
- Trivyとgovulncheckの結果をアップロードできる
- 前回との差分が正しく確認できる
- Slack通知が実用に耐える
- CI成果物を直接見るより便利だと感じられる
- 3〜5人の外部ユーザーに試してもらえる
- 1チーム以上が継続利用したいと言う

## 10. 初期課金方針

MVP時点では、課金実装は必須ではない。

ただし、将来的な課金を見据えて、プラン設計は意識する。

想定プランは以下。

- Free：1プロジェクト、履歴30日
- Solo：3プロジェクト、履歴90日
- Team：10プロジェクト、履歴1年、Slack通知

MVPでは、まず無料βとして提供し、利用継続と支払い意欲を検証する。

## 11. MVPの優先順位

### 必須

- アカウント作成
- プロジェクト作成
- APIキー発行
- Trivy JSONアップロード
- govulncheck JSONアップロード
- スキャン履歴一覧
- 重要度別集計
- Finding一覧
- 前回との差分表示
- Slack通知

### 準必須

- Finding詳細
- Ignore設定
- Discord通知
- GitHub Actions用サンプル
- GitLab CI用サンプル

### 後回し

- グラフ表示
- Stripe課金
- チーム招待
- 複雑な権限管理
- GitHub App連携
- 自動Issue作成
- AI修正提案

## 12. 最初に作るべき最小版

最初の最小版は、以下までに絞る。

1. プロジェクトを作れる
2. APIキーを発行できる
3. Trivy JSONをアップロードできる
4. 最新スキャン結果を見られる
5. 前回との差分を見られる
6. Slack通知できる

govulncheck対応、Finding詳細、Ignore設定はその次でよい。

## 13. MVPの一文定義

FindingDeckのMVPは、**CIで生成されたOSSセキュリティスキャナのJSONレポートを受け取り、重要度・差分・履歴・通知としてチームが扱える状態にする最小プロダクト**である。
